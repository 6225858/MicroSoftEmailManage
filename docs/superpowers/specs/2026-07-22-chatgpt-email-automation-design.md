# ChatGPT 邮箱自动领取与验证码接入设计

日期：2026-07-22

## 1. 背景

`pix-automation-2.4.3` 已具备 ChatGPT 登录页面自动填写、验证码轮询、验证码自动输入和 Session Token 提取能力，但当前验证码客户端依赖另一套会接收 Outlook `client_id` 与 `refresh_token` 的取件协议。

`MicroSoftEmailManage` 已集中保存 Microsoft 邮箱凭据、标签和邮件缓存，并提供 API Key 机制。此次改造让插件从该服务自动领取一个可注册邮箱，由服务端读取验证码，并在注册完成后标记邮箱。邮箱密码、OAuth Client ID、Refresh Token 和邮件正文都不离开邮箱服务。

## 2. 目标

- 插件不再要求用户粘贴四段 Outlook 账号串。
- 插件自动领取一个不含精确标签 `已注册chatgpt` 的有效邮箱。
- 多个插件实例不能同时领取同一邮箱。
- 邮箱服务只接受带有效 `X-Api-Key` 的自动化请求。
- 邮箱服务从 Inbox 和 Junk 中严格识别本次登录对应的 ChatGPT 验证邮件。
- 插件取得验证码后自动填写并继续登录。
- 插件取得 Session Token 后，邮箱服务为账号去重追加 `已注册chatgpt` 标签。
- 未完成任务可主动释放，异常中断的占用可超时回收。
- 完成通知可安全重试，避免已经注册成功的邮箱因响应丢失而被再次领取。

## 3. 非目标

- 不自动创建或导入 Microsoft 邮箱账号。
- 不向插件返回邮箱密码、OAuth Token、邮件正文或完整账号列表。
- 不改变 `MicroSoftEmailManage` 现有管理页面和既有 API 的宽松认证行为；只有新增自动化接口强制 API Key。
- 不扩展为通用验证码平台；只识别本设计定义的 ChatGPT 验证邮件。
- 不在测试中访问真实 Microsoft 邮箱或提交真实 ChatGPT 注册。

## 4. 已选方案

采用专用自动化 API 和持久化租约表。插件只持有领取结果与随机领取 Token；邮箱服务负责账号选择、并发占用、邮件读取、验证码提取和最终标签更新。

未采用的方案：

- 临时标签：会污染业务标签，异常退出后容易遗留，并发控制较弱。
- 插件组合现有账号与邮件接口：会暴露不必要的账号和邮件信息，也无法可靠防止重复领取。
- 兼容旧 `/api/graph/mail-all` 协议：会保留不再需要的 OAuth 凭据字段和错误的职责边界。

## 5. 数据模型

新增表 `chatgpt_email_claim`：

| 字段 | 类型 | 约束 | 用途 |
| --- | --- | --- | --- |
| `id` | Integer | 主键 | 内部标识 |
| `mail_account_id` | Integer | 唯一、非空、索引 | 被领取的邮箱账号 |
| `claim_token` | Text | 唯一、非空、索引 | 随机领取凭证，至少 32 字节熵 |
| `status` | Text | 非空 | `active` 或 `completed` |
| `claimed_at` | Integer | 非空 | Unix 秒时间戳 |
| `expires_at` | Integer | 非空、索引 | 活跃租约过期时间 |
| `completed_at` | Integer | 默认 0 | 完成时间；完成回执保留期依据 |

`mail_account_id` 的唯一约束是并发安全的最终防线。领取操作在 SQLite 写事务中清理过期记录、选择账号并插入租约；若发生唯一约束竞争，事务重试选择其他账号。

普通活跃租约有效期为 15 分钟。每次有效验证码查询将活跃租约续期到当前时间之后 15 分钟。找到验证码后，租约延长到当前时间之后 24 小时，降低注册已完成但完成通知暂时失败时被重复领取的风险。

完成时，记录转为 `completed`，不再视为活跃租约。完成回执保留 24 小时，使相同 `claim_token` 的重试得到幂等成功响应；之后可清理记录。账号上的 `已注册chatgpt` 标签是长期完成依据。

主动释放会删除尚未完成的租约。已完成记录不能通过释放接口撤销。

## 6. 标签与账号资格

标签使用现有逗号分隔格式解析，同时兼容中文逗号。判断和写入均基于去除首尾空白后的完整标签，不使用 SQL 子串或字符串 `includes` 判断。

账号满足以下全部条件才可领取：

- `valid_status == 1`；
- 不含精确标签 `已注册chatgpt`；
- 没有未过期的 `active` 租约；
- 没有 `completed` 回执记录。

服务端从所有符合条件的账号中选择一个。具体顺序不构成 API 契约，测试不得依赖固定邮箱顺序。

完成操作保留原标签顺序与内容，去重追加 `已注册chatgpt`。重复完成不会重复写入标签。

## 7. 自动化 API

所有接口路径以邮箱服务根地址为基准，并强制要求：

```http
X-Api-Key: ms_example
Content-Type: application/json
Cache-Control: no-store
```

缺少或传入无效 API Key 均返回 `401`。响应和服务日志不得输出 API Key、领取 Token、验证码或邮箱凭据。

### 7.1 领取邮箱

```http
POST /api/automation/chatgpt/claims
```

无需请求体。成功响应：

```json
{
  "email": "user@example.com",
  "claim_token": "opaque-random-token",
  "expires_at": 1784700000
}
```

没有可用邮箱时返回：

```json
{
  "detail": {
    "code": "no_available_email",
    "message": "没有可用邮箱"
  }
}
```

HTTP 状态为 `409`。插件收到该响应后不打开登录页。

### 7.2 查询验证码

```http
POST /api/automation/chatgpt/verification-code
```

请求体：

```json
{
  "claim_token": "opaque-random-token",
  "not_before": 1784699000000
}
```

`not_before` 是插件开始本次登录时记录的 Unix 毫秒时间戳。服务端允许向前 120 秒的时间容差，以兼容邮件时间精度和主机时钟微小偏差。

找到验证码时返回：

```json
{
  "code": "919020",
  "received_at": "2026-07-22T13:48:45+08:00",
  "folder": "inbox"
}
```

尚未找到匹配邮件时返回 HTTP `200`：

```json
{
  "code": "",
  "received_at": "",
  "folder": ""
}
```

未知领取 Token 返回 `404`，过期领取返回 `410 claim_expired`，已完成领取返回 `409 claim_completed`。邮箱刷新明确失败且没有可用新缓存时返回 `502 mail_fetch_failed`。

### 7.3 完成注册

```http
POST /api/automation/chatgpt/claims/complete
```

请求体：

```json
{
  "claim_token": "opaque-random-token"
}
```

服务端在同一数据库事务中为邮箱去重追加 `已注册chatgpt`，并将领取记录转为 `completed`。首次调用和回执保留期内的重复调用均返回：

```json
{
  "ok": true,
  "status": "completed"
}
```

### 7.4 释放邮箱

```http
POST /api/automation/chatgpt/claims/release
```

请求体：

```json
{
  "claim_token": "opaque-random-token"
}
```

活跃领取被删除后返回 `{ "ok": true, "released": true }`。重复释放未知或已删除的 Token 返回 `{ "ok": true, "released": false }`，便于清理操作幂等。已完成领取返回 `409 claim_completed`，不能通过释放撤销注册标签。

## 8. 邮件刷新与严格筛选

验证码接口复用现有邮件缓存与去重刷新能力：每次查询为对应账号的 `inbox` 和 `junk` 触发后台刷新，并检查当前缓存。插件后续轮询会读取刷新后的缓存，不为每个 2.5 秒轮询重复创建 Microsoft 请求。

一封邮件必须同时满足以下条件：

1. `mail_from` 中解析出的地址忽略大小写后精确等于 `noreply@tm.openai.com`；
2. `subject` 去除首尾空白并忽略大小写后精确等于 `Your temporary ChatGPT verification code`；
3. `mail_to` 中解析出的地址包含当前领取邮箱，忽略大小写；
4. 邮件时间不早于 `not_before - 120 秒`；
5. 正文包含固定语句 `Enter this temporary verification code to continue:`，其后第一个独立数字必须恰好为六位。

地址从展示格式中解析，例如：

```text
ChatGPT (noreply@tm.openai.com)
user@outlook.com (user@outlook.com)
```

正文可能为 HTML。服务端使用结构化 HTML 文本提取方式移除标签、解码实体并归一化空白，然后以固定语句为锚点提取六位验证码。SendGrid 跳转链接及正文其他位置的数字不参与候选匹配。

Inbox 和 Junk 中若有多封满足条件的邮件，选择邮件时间最新的一封。无有效时间的邮件不参与自动填写。

## 9. 插件改造

### 9.1 设置

验证码接口设置调整为：

- 邮箱服务地址，默认 `http://192.168.1.27:10019/`；
- API Key，使用密码输入框显示。

两项保存在 `chrome.storage.local`。插件日志、状态消息和错误消息不得回显 API Key。浏览器配置文件的本地存储不是硬件级秘密存储，使用者仍需保护浏览器配置文件。

启动任务前，插件根据服务地址申请 `${origin}/*` 可选 Host Permission。拒绝授权时停止任务。

### 9.2 登录任务

原账号串输入改为只读的“当前领取邮箱”显示。用户点击“开始注册登录”后：

1. 校验服务地址与非空 API Key；
2. 请求 Host Permission；
3. 调用领取接口；
4. 将 `email`、`claim_token`、`startedAt` 和任务状态写入扩展本地任务；
5. 打开 ChatGPT 登录页并填写邮箱。

插件不再解析或向邮箱服务发送密码、`client_id`、`refresh_token`。

### 9.3 验证码

登录页进入验证码阶段后，每 2.5 秒向后台服务工作线程请求验证码。后台线程从设置中读取服务地址和 API Key，并调用专用验证码接口；API Key 不传入网页上下文。

返回空验证码时继续等待。返回六位验证码后停止轮询，填写验证码并继续登录。手动验证码输入保留为异常情况下的人工兜底，但手动填写不会绕过领取与完成标记生命周期。

### 9.4 完成与恢复

取得 Session Token 是“注册成功”的唯一客户端触发点。后台服务工作线程立即调用完成接口。

若完成调用失败：

- 保存 `pending_completion` 状态；
- 不调用释放接口；
- 当前运行期间重试；
- 使用 `chrome.alarms` 定期唤醒后台线程补交；
- 浏览器或扩展重启后继续补交；
- 成功或服务端返回幂等完成后清除待完成状态。

因此插件清理按钮执行以下分支：

- 尚未取得 Session Token：调用释放接口，然后清理本地任务；
- 已取得 Session Token 或处于 `pending_completion`：先补交完成，不释放邮箱。完成仍失败时保留最小待完成状态并向用户显示原因。

Manifest 增加 `alarms` 权限，用于可靠重试完成通知。

## 10. 错误处理

| 场景 | 服务端行为 | 插件行为 |
| --- | --- | --- |
| API Key 缺失或错误 | `401` | 立即停止，不重试 |
| 无可用邮箱 | `409 no_available_email` | 提示并停留侧栏，不打开登录页 |
| 领取 Token 未知 | `404` | 停止当前自动化并提示重新领取 |
| 租约过期 | `410 claim_expired` | 停止填写并提示重新领取 |
| 暂无验证码 | `200`，空 `code` | 继续轮询，日志限频 |
| 邮件刷新失败 | `502 mail_fetch_failed` | 保留任务，限频提示并继续尝试 |
| 完成通知失败 | `5xx` 或网络错误 | 持久化待完成状态并重试，禁止释放 |
| 用户清理未完成任务 | 删除活跃租约 | 清理本地任务 |

操作日志继续使用现有脱敏机制，并补充对领取 Token、API Key 和六位验证码的覆盖。邮箱明文只显示在当前领取邮箱区域，不写入持久化操作日志。

## 11. 测试设计

### 11.1 邮箱服务测试

- 只领取不含精确标签 `已注册chatgpt` 的 `valid_status == 1` 账号；
- 相似标签不会被误判为精确标签；
- 已注册、无效、有有效租约的账号不会被领取；
- 并发领取无法获得同一账号；
- 过期租约可回收，未过期租约不可重复领取；
- 完成操作保留原标签并去重追加目标标签；
- 完成响应丢失后的重试得到幂等成功；
- 释放只删除对应活跃领取，不能撤销完成状态；
- 自动化 API 对缺失和错误 API Key 均返回 `401`。

### 11.2 邮件筛选测试

- 给定确认过的发件人、标题、收件人、时间和正文，提取 `919020`；
- 发件人、标题或收件人任一不符时拒绝；
- 登录开始时间之前的邮件拒绝；
- 非六位数字拒绝；
- 固定语句之外的数字和 SendGrid 链接数字不会被识别；
- HTML 正文经解析后仍能识别固定语句和验证码；
- Inbox 与 Junk 同时匹配时选择时间最新的邮件；
- 无有效时间的邮件不参与自动填写。

测试使用临时 SQLite 数据库和模拟邮件缓存/刷新结果，不读取真实 `mail.db`。

### 11.3 插件测试

- 服务地址和 API Key 的保存、加载与校验；
- 默认服务地址为 `http://192.168.1.27:10019/`；
- 领取成功后创建只含必要字段的登录任务；
- 插件不发送邮箱密码、Client ID 或 Refresh Token；
- `401`、无可用邮箱、租约过期、暂未收到验证码进入各自状态；
- 六位验证码能传给页面自动填写；
- Session Token 成功后调用完成接口；
- 完成失败会保存待重试状态，后台唤醒或重启后继续补交；
- 清理未完成任务会释放，待完成任务不会释放；
- 日志不包含 API Key、领取 Token、验证码或邮箱明文。

### 11.4 验证命令范围

- Python 测试套件；
- 插件现有及新增 Node 测试；
- 所有插件 JavaScript 文件的 `node --check`；
- 使用模拟 API 与模拟登录页执行浏览器烟雾测试。

## 12. 文档与兼容性

- 更新两个项目的 README，说明新配置、API、标签生命周期和局域网部署要求。
- 插件旧取件 API 设置键可在首次加载时迁移或删除，但不得把旧 OAuth 凭据带入新任务。
- 现有手动验证码输入、Session Token 展示和后续支付流程继续使用当前领取邮箱。
- 邮箱服务现有账号管理、标签管理和邮件浏览接口保持兼容。

## 13. 验收标准

在至少两个并行插件任务的模拟环境中：

- 两个任务领取到不同的、未带 `已注册chatgpt` 标签的有效邮箱；
- 每个任务只能读取自己领取邮箱的严格匹配 ChatGPT 验证码；
- 验证码自动填写后可继续登录；
- Session Token 到达后对应邮箱被精确添加 `已注册chatgpt` 标签；
- 完成响应丢失并重试不会重复标签或重新发放邮箱；
- 未完成任务释放或超时后邮箱可再次领取；
- API 和日志不泄露邮箱凭据、邮件正文、API Key、领取 Token 或验证码。
